# pdf2pptx

PDFを編集可能なPPTXに変換するCLIツール（自社ツール・Phase 1.2）。
CLI（`convert.py`）に加えて、ブラウザからアップロードして変換・ダウンロードできる
**ローカル専用のWebアップロードMVP**（`app.py`）も用意している。

## 方式

1. PyMuPDFでテキストを span / line 単位で抽出する（座標・フォントサイズ・色・太字/斜体）
2. 編集対象テキストの領域を **redaction で背景から削除**する（編集テキストとの二重写りを防止。
   画像・罫線・図形・表の枠は残る）
3. redaction後のページを画像化（既定150dpi）してスライド背景いっぱいに敷く
4. 抽出テキストを同じ座標に**編集可能なテキストボックス**として重ねる
   （PDF pt → PPTX EMU 変換。スライドサイズ＝PDFページサイズ。内部マージン0・折り返しなし）

Phase 1 の範囲:

- 編集対象は**横書き・可視のテキストのみ**。次はすべて背景画像側に残す（表示は保たれるが編集不可）:
  - 縦書き・回転テキスト（wmodeや書字方向ベクトルで判定）
  - **ページ自体が回転している場合**（/Rotate）。座標補正はPhase 2以降のため、
    そのページは丸ごと背景画像のみにフォールバックする。
  - **不可視テキスト**（`alpha=0` または PDFの描画モード3=invisible）。
    OCRソフトがスキャン画像の上に重ねる検索用テキスト層で典型的に使われる手法。
    これを誤って編集可能テキスト化すると、画像の上に見えるはずのないテキストが
    可視の状態で二重に出てしまうため、必ず除外する。
  - 文字コードを復元できないテキスト（ToUnicode欠落等）
  - 上記いずれかと bbox が重なる横書きテキスト。redactionでその領域を消すと
    重なっている縦書き/回転/不可視テキストまで一緒に欠損してしまうため、
    重なりを検出した行は編集対象にせず両方とも背景に残す。
  - **ページ面積の大部分（既定85%以上）が画像で覆われているページの可視テキスト**。
    画像に焼き込まれた文字を編集可能テキスト化すると、画像と重なって二重写りに
    なるリスクがあるため、OCRの不可視テキスト層でなくても丸ごと背景画像に残す
    （閾値は `FULL_IMAGE_COVERAGE_THRESHOLD` 定数）。
- スキャンPDF（テキスト層なし、または不可視OCRテキスト層のみ）は検出し、
  背景画像のみのスライド+警告を出す。
- ページサイズ混在PDFは警告を出し、1ページ目基準のスライドに中央配置で収める。
  縮小はしても**拡大はしない**（小さいページが不必要に引き伸ばされない）。
- フォントは簡易マッピング（ＭＳ ゴシック等の対応表 + 和文/欧文・セリフ判定によるフォールバック）。
  日本語には `a:ea`（東アジアフォント）も設定する。
- 入力PDFと出力PPTXに同じパスを指定した場合はエラーで停止する（`resolve()`の一致に加え、
  出力が既に存在する場合は `os.path.samefile()` でハードリンクも検出して拒否する）。
- 入力ファイルの先頭に `%PDF-` マジックバイトがあるか、`doc.is_pdf` が真かを確認し、
  PDFでないファイルは早期に拒否する。
- 1ページ目のサイズが PowerPoint のスライドサイズ制限（1〜56インチ）を外れる場合は拒否する
  （スライドサイズはPDFの1ページ目基準で決まるため）。
- dpiや各種上限値に0以下の値が渡された場合は拒否する。
- Web公開等を見据え、ページ数・dpi・背景画像の画素数（1ページ/全ページ合計）・
  入出力ファイルサイズ・処理時間に上限を設けており、超過時は安全に停止する
  （既定値は下記オプション参照）。全ページ合計の画素数は実際に描画する前に検査し、
  各ページの背景画像も溜め込まずに都度スライドへ追加してから破棄することで
  ピークメモリ使用量を抑えている。

## セットアップ

Python 3.10 以上が必要（このマシンでは uv を使う）:

```bash
uv venv --python 3.12 .venv
uv pip install --python .venv/bin/python -r requirements.txt
```

## 使い方

```bash
.venv/bin/python convert.py input.pdf output.pptx
```

オプション:

- `--dpi N` … 背景画像の解像度（既定: 150）
- `--debug-dir DIR` … redaction後の背景PNGを保存する**空のディレクトリ**（検証用）。
  既存ファイルがあるとエラーで停止する（上書き・混在事故防止）。
  **機密PDFや本番運用では使わないこと** — 背景画像が平文PNGとしてディスクに残る。
  途中でエラー・タイムアウトになった場合、それまでに処理したページ分のPNGだけが
  残る（不完全な集合になりうる）ため、成果物としてではなく都度の検証用途に限る。
- `--max-pages N` … ページ数の上限（既定: 200）
- `--max-dpi N` … dpiの上限（既定: 300）
- `--max-page-pixels N` … 背景画像1ページあたりの画素数上限（既定: 約50,000,000＝50MP）
- `--max-total-pixels N` … 背景画像の全ページ合計の画素数上限（既定: 約500,000,000＝500MP）
- `--max-file-size-mb N` … 入力PDFのファイルサイズ上限MB（既定: 100）
- `--max-output-size-mb N` … 出力PPTXのファイルサイズ上限MB（既定: 300。超過時は出力ファイルを削除する）
- `--timeout SECONDS` … 処理時間の上限秒（既定: 無制限。**ソフトタイムアウト** — 詳細は後述）

上限を超えた場合は例外（`ValueError` / `TimeoutError`）で処理を停止する。
CLIではエラーメッセージを表示して終了コード1を返す。

縦書き・回転ページ・不可視OCRテキスト・全面画像ページ・スキャンページ等の警告は
標準エラーに出力される。

### Web公開時のハードタイムアウト（worker.py）

`convert.py` の `--timeout` はプロセス内で「ページ処理の合間」にしかチェックできない
**ソフトタイムアウト**であり、1ページの描画がPyMuPDF等のネイティブコード内でハング
した場合には効かない。Webアプリ等、外部からの入力を確実な時間内に打ち切る必要が
ある場合は `worker.py` を使う。`convert.py` を別プロセスとして起動し、親プロセスから
**hard_timeout秒でプロセスグループごとSIGKILL**する（OSレベルの強制終了のため、
ネイティブコード内でハングしていても確実に停止できる。`start_new_session=True` で
子を新しいプロセスグループのリーダーにし、タイムアウト時は `os.killpg()` でグループ
全体を強制終了するため、子プロセスがさらに孫プロセスを起動していても道連れにできる）。

出力は一時ディレクトリに作らせ、子プロセスが正常終了した場合のみ「出力先と同じ
ディレクトリの `.part` ファイル」へコピーし、zipとして開けるかを検証したうえで
`os.replace()` により最終出力へ**アトミックに反映**する。`shutil.move()` は入力元
（一時ディレクトリ、通常 `/tmp` 配下）と出力先が別ファイルシステムの場合にコピー+
削除の非アトミックな動作にフォールバックし、途中終了時に不完全なファイルが出力先に
残りうるため使わない。検証に失敗した場合や子プロセスが失敗した場合、`.part` ファイル
は残さず削除する。一時ディレクトリの削除も試みるが `ignore_errors=True` で完全に
黙殺はせず、削除に失敗した場合はstderrに警告を残す（本番運用では、削除に失敗した
一時ディレクトリを定期的に掃除するjanitorプロセスの用意を推奨する）。

worker.py自身も、実行前に入力と出力が同一ファイルでないことを検査する
（`resolve()`の一致に加え `os.path.samefile()` でハードリンクも検出）。
convert.py側にも同じ検査があるが、worker.pyは子プロセスに実際の出力先ではなく
一時ディレクトリ内のパスを渡すため、convert.py側の検査だけではworker.py経由の
実行を保護できない。危険な最終書き込み（`.part`→最終出力への置換）はworker.py
（親プロセス）側で行われるため、worker.py自身がこの検査を持つ必要がある。

```bash
.venv/bin/python worker.py input.pdf output.pptx --hard-timeout 120 [convert.pyと同じオプション]
```

Webアプリからは `worker.run_with_hard_timeout(input_pdf, output_pptx, hard_timeout=...)`
を直接呼び出すこともできる。

### Web公開時のリソース隔離方針

**ハードタイムアウトだけでは不十分。** worker.pyは「時間」の暴走は止められるが、
「RAM・CPU・ディスク・出力サイズ」の暴走はOS/コンテナ側の隔離がなければ止められない
（例: タイムアウトに達する前に大量のメモリを確保し尽くす、一時ディスクを埋め尽くす等）。
本番でWebから利用する場合は、必ず次の方針を満たすこと。

- **プロセス隔離**: 変換ジョブはコンテナ（Docker等）またはcgroupで実行し、
  RAM上限・CPU上限・一時ディスク(tmpfsやディスククォータ)上限を必ず設定する。
  本ツールの `--max-*` オプションはアプリケーションレベルの目安に過ぎず、
  バグや未知の入力パターンに対する最後の防衛線にはならない。
- **非特権・ネットワークなしワーカー**: 変換を実行するワーカーは非rootユーザーで
  実行し、ネットワークアクセスを持たせない（PyMuPDF等のライブラリに未知の
  脆弱性があった場合の被害範囲を、外部への通信路を断つことで限定するため）。
- **ユーザー指定パスの禁止**: Webサービスの利用者に `input`/`output` の実ファイル
  パスや `--debug-dir` を直接指定させてはならない。パスは常にサービス側が
  ジョブごとに払い出す一時パス（例: ジョブIDベースのUUIDパス）に固定し、
  利用者からの入力はアップロードされたバイト列としてのみ受け取る。
  `--debug-dir` は本番では常に無効（未指定）にする。
- **一時ファイルの掃除**: worker.pyは一時ディレクトリの削除を試みるが、
  プロセスが強制終了された場合（例: コンテナ自体がOOM killされた場合）は
  そもそもPythonのfinally/exceptすら実行されず削除できない。定期的に
  古い一時ファイルを掃除するjanitor（cronやコンテナの再起動時クリーンアップ等）
  を別途用意すること。

## Webアップロード MVP（app.py）

PDFをブラウザからアップロードし、`worker.py`（ハードタイムアウト付き別プロセス）
経由でPPTXに変換し、完了後にダウンロードできるFastAPI製のローカル専用Webアプリ。
**認証・決済・S3等のクラウド保存・同時実行数の制限・本番公開は一切実装していない
MVPであり、このマシン上でローカルに動かす前提**。本番公開する場合は上記
「Web公開時のリソース隔離方針」を満たすこと。

### ローカルでの起動方法

```bash
uv pip install --python .venv/bin/python -r requirements.txt   # fastapi/uvicorn等を含む
.venv/bin/uvicorn app:app --host 127.0.0.1 --port 8000 --reload
```

ブラウザで `http://127.0.0.1:8000/` を開くとアップロード画面が表示される。
PDFを選択（またはドラッグ&ドロップ）して「アップロードして変換」を押すと、
自動的に状態をポーリングし、完了したらダウンロードリンクが表示される。

### エンドポイント

| メソッド/パス | 内容 |
|---|---|
| `GET /` | アップロード用の簡易HTML画面 |
| `POST /upload` | PDFをアップロードして変換ジョブを開始する（`multipart/form-data`、フィールド名 `file`）。`202` + `{"job_id": ..., "status": "queued"}` を返す |
| `GET /jobs/{job_id}` | ジョブの状態(`queued`/`processing`/`done`/`error`)・警告・エラーメッセージをJSONで返す |
| `GET /jobs/{job_id}/download` | 完了(`done`)したジョブのPPTXをダウンロードする。未完了なら`409`、ジョブが無ければ`404` |

### 設計上の安全策

- **ユーザー指定パスを一切受け取らない。** 保存先は常にサーバーが生成する
  UUID（`jobs/<job_id>/`）配下の固定名（`input.pdf` / `output.pptx`）。
  アップロードされたファイルの元のファイル名はダウンロード時の表示名
  （`Content-Disposition`）にのみ使い、保存パスの構築には使わない。
- **job_idはURLパスから来るため、ファイルパスに使う前に必ずUUIDとして検証する**
  （パストラバーサル対策。UUIDとして解釈できない値は`404`にする）。
- **アップロードの検証**: 拡張子(`.pdf`)・ファイルサイズ上限
  （既定100MB、ストリーミング読み込みで超過を即座に検出）・`%PDF-`マジック
  バイトを確認する。いずれかに失敗したジョブディレクトリは残さず削除する。
- **ダウンロード直前の出力ファイル検証。** `worker.py`の正常なアトミック
  置換（`.part`→`os.replace()`）では起こらないはずだが、万一の細工・競合・
  バグに備え、`GET /jobs/{job_id}/download` は返す直前に必ず次を確認する
  （`app._validate_output_for_download()`）。いずれかに失敗した場合は
  `404`/`500`どちらであっても、内部パスやsymlinkの指す先を含む詳細は
  サーバーログにのみ出し、レスポンスには固定の安全な文言しか返さない。
  - `output.pptx` が symlink でないこと
  - `output_path.resolve(strict=True)` がジョブディレクトリの
    `resolve()` 配下であること
  - `zipfile.is_zipfile()` で開けること（pptxはzip形式のため）
- **エラーメッセージのパス秘匿。** 変換失敗時、`worker.py`の標準エラー出力
  （一時ディレクトリ等の内部パスを含みうる）はそのままAPI/UIに出さない。
  `GET /jobs/{job_id}` が返す`message`は常に固定の安全な文言
  （`app.GENERIC_CONVERSION_ERROR_MESSAGE`）にし、生の詳細は
  `status.json`の`internal_error`フィールドにのみ記録する。
  `internal_error`はAPIレスポンスから除外する（`app._public_status()`）。
- **ファイル名の長さ制限。** アップロード元のファイル名は保存時に255文字
  程度（拡張子は保持）へ、ダウンロード時に提示するファイル名はstemを
  180文字程度へ切り詰め、`Content-Disposition`ヘッダが肥大化しないようにする。
- **debug-dirはWeb経由では常に無効。** `worker.run_with_hard_timeout()` を
  呼ぶ際、`debug_dir` は常に `None` を渡す（コード上ユーザーが指定する
  手段自体が無い）。
- **変換は必ずworker.py（ハードタイムアウト付き別プロセス）経由。** `convert.py`
  を直接同一プロセスで呼び出すことはしない。
- バックグラウンドタスク内で例外が起きた場合も必ず`error`状態に遷移させる
  （そうしないとジョブが`processing`のまま無限に見えてしまうため）。

### ジョブディレクトリと掃除（janitor.py）

```
jobs/
  <job_id>/
    input.pdf        # アップロードされたPDF
    output.pptx       # 変換結果（成功時のみ）
    status.json       # {"status": ..., "created_at": ..., "warnings": [...],
                       #  "internal_error": ...（API/UIには出さない）, ...}
```

各ジョブは自己完結した1ディレクトリなので、一定時間より古いものをまとめて
削除できる。`janitor.py` がその実装で、cronや手動実行を想定している。

```bash
.venv/bin/python janitor.py --max-age-hours 24 --stale-timeout-hours 48
```

`janitor.py`の安全策:

- **queued/processingは特別扱いする。** 変換がまだ実行中の可能性がある
  ジョブを誤って消さないよう、`queued`/`processing`のジョブは通常の
  `--max-age-hours`ではなく、より長い`--stale-timeout-hours`（既定は
  `--max-age-hours`と同じ）を超えた場合のみ「放置されたジョブ」とみなして
  削除する。`done`/`error`（変換が終わったジョブ）はこれまで通り
  `--max-age-hours`で削除する。
- **不正・不一致なエントリはskipする。** ディレクトリ名がUUID形式でない
  もの、`status.json`内の`job_id`がディレクトリ名と一致しないもの
  （破損・改変・別プロセスとの競合の疑い）は削除対象にせず、次回以降の
  実行に委ねる。
- **削除失敗を黙殺しない。** `shutil.rmtree(ignore_errors=True)`は使わず、
  削除に失敗したジョブは警告として標準エラー出力に残し、CLIの終了コードも
  `1`にする（cron等の監視で検知できるようにするため）。

`app.py` 自体は自動でジョブディレクトリを削除しない（ダウンロードのため）。
本番相当の運用をする場合は `janitor.py` を定期実行すること（README冒頭の
「Web公開時のリソース隔離方針」の一時ファイル掃除の項も参照）。

### 本番公開までに追加で必要なもの（このMVPには無い）

このMVPはローカルでの動作確認を目的としており、次は意図的に実装していない。
本番公開する場合は必ず用意すること。

- **reverse proxy / ASGI middlewareでのリクエストボディサイズ制限。**
  `app.py`のアップロードサイズ上限（既定100MB）はアプリケーション層の
  チェックに過ぎない。悪意あるクライアントがContent-Lengthを詐称したり、
  極端に遅く送り続けたりする攻撃に備え、nginx等のreverse proxyや
  ASGIミドルウェアでも上限を設定すること。
- **BackgroundTasksは公開用のジョブキューではない。** FastAPIの
  `BackgroundTasks`はアプリプロセス内のスレッドプールで実行されるだけで、
  永続化・再試行・分散実行・プロセスクラッシュ時の復旧機能が無い。
  プロセスが落ちれば実行中のジョブも失われる。公開版では
  RQ / Celery / Arq 等の外部ジョブキューを導入し、Webプロセスと
  変換ワーカーを別プロセス・別コンテナに分離すること（Phase 0.3以降）。
- **レート制限・認証・同時変換数の制限・jobs総容量の制限が無い。**
  誰でも無制限にアップロードでき、同時に何本でも変換ジョブを起動でき、
  `jobs/`ディレクトリの総容量にも上限が無い。公開する場合は最低限、
  IP単位のレート制限、何らかの認証、同時実行中のジョブ数の上限、
  `jobs/`全体のディスク使用量上限を追加すること。

## テスト

```bash
.venv/bin/python -m pytest -q
```

フィクスチャは `tests/fixtures_gen.py` が生成する合成PDFのみ（ネットワーク不使用）。
表・画像・色・太字・回転テキスト・スキャン風ページ・ページサイズ混在（縮小・拡大なし双方）・
/Rotate付きページ・不可視OCRテキスト層・重なり合うテキスト・画像多数のページ・全面画像+
可視テキスト・各種上限超過（1ページ/全ページ合計画素数・出力サイズ・非正値）・
PDFでないファイル・極端なページサイズ（大小とも）・入出力パス同一（resolve一致・
ハードリンク）・debug-dirの既存ファイル拒否・worker.pyのハードタイムアウト（プロセス
グループごとのkillで孫プロセスも道連れにできること含む）・worker.py自身での入出力
パス同一拒否（resolve一致・ハードリンク）・`.part`ファイルによるアトミックな出力反映
（成功時の置換・失敗時の残留なし）を網羅する。`tests/test_app.py` はFastAPIの
TestClientでアップロード〜変換〜ダウンロードの一連の流れ・入力検証・パストラバーサル
対策・debug_dirが常にNoneで渡されること・ダウンロード直前の出力ファイル検証

内部パスが漏れないこと・ファイル名の長さ制限を検証する。`tests/test_janitor.py` は

起動せず、ジョブディレクトリはテストごとに一時パスへ隔離する）。

## 実PDFでの検証

合成PDF（テスト用フィクスチャ）だけでなく、実務で使われている書類（請求書・
見積書・契約書・Excel/Word由来の表・スキャンPDF等）10件で変換品質を検証済み。
結果は `docs/real_pdf_test_log.md` を参照（取引先名・個人名等は匿名化済み）。

- **○ 完全変換: 7件 / △ 一部制限あり: 3件 / × 実用不可: 0件**
- 二重写り（redaction漏れ）は全件で0件
- △の3件はすべてPhase 1の設計上の安全側フォールバック（ページ回転・スキャン
  検出）によるもので、レイアウト破綻や欠損によるものではない
